# m0.8 双向消息通知机制 (Schema 变更 + 业务上报)

**业务项目 ↔ PMO 双向通知，完整 Pull + Push 混合方案**

## 核心原则

```
PMO 变更 → 业务系统拉取（Pull 为主）
业务系统上报 → PMO 接收（Push 到 Message-Broker）
```

- **PMO 不主动 Push 内容**，只发轻量通知（Ping），内容由业务系统主动拉取
- **业务系统主动 Push 指标/事件/异常**到 PMO Message-Broker
- PMO 不依赖业务系统在线，业务系统自控同步节奏
- 所有消息经 PMO 中介，可监控 + 可审计

---

## 方向一：PMO → 业务系统（Schema 变更通知）

### 1. 触发机制

**变更监听器**（PMO 侧 `scripts/runtime/schema_watcher/`）：

```
监听文件:
  - config/biz-meta/E2-schema-{id}.json    # 业务数据 schema
  - config/biz-meta/E3-glossary-{id}.json  # 业务术语表
  - config/data-sync.yaml                  # 数据同步策略

触发条件（满足任一）:
  - 文件内容 hash 发生变化（sha256）
  - 文件版本号字段（version）递增
  - data-sync.yaml 中 sync_policy 变更

变更事件结构:
{
  "event_id": "EVT-{timestamp}-{seq}",
  "event_type": "schema_change",
  "changed_files": [
    {
      "file": "E2-schema-finance.json",
      "old_hash": "abc123",
      "new_hash": "def456",
      "old_version": "1.0",
      "new_version": "1.1",
      "changed_at": "2026-06-19T10:00:00Z"
    }
  ],
  "affected_biz_projects": ["1.2-finance", "1.3-hedge-fund"],
  "severity": "info" | "breaking" | "critical",
  "timestamp": "2026-06-19T10:00:00Z"
}
```

### 2. 通知机制（轻量 Ping）

PMO 通过 Message-Broker 发送轻量通知（**不发完整内容，只发事件摘要**）：

```
主题: pmo.schema.change
类型: notification
QoS: 1 (at-least-once)
内容: { event_id, event_type, changed_files_summary, severity, pmo_instance }

业务系统订阅后:
  1. 收到 Ping 通知（轻量）
  2. 根据 event_id + changed_files 调用 PMO GET 接口拉取完整变更
  3. 拉取后 ACK（可选，QoS 1 自动 ACK）
```

### 3. 业务系统拉取接口

```
GET /pmo-api/v1/schema/changes/{event_id}
Response: {
  "event_id": "EVT-20260619-001",
  "changed_files": [...],
  "diff": { ... },          # schema diff（可选项）
  "migration_guide": "...", # 破坏性变更时提供迁移提示
  "deadline": "2026-06-26", # 业务系统适配截止日期（breaking 时）
  "issued_by": "PMO-L1",
  "issued_at": "2026-06-19T10:00:00Z"
}

GET /pmo-api/v1/schema/changes?since={timestamp}&biz_project={id}
Response: {
  "changes": [...],
  "total": 5,
  "page": 1,
  "per_page": 20
}
```

### 4. ACK 确认机制

```
业务系统适配完成后，通过 Message-Broker 发送确认：

主题: biz.{biz_project_id}.schema.ack
类型: notification
内容: {
  "event_id": "EVT-20260619-001",
  "biz_project_id": "1.2-finance",
  "status": "adapted" | "in_progress" | "blocked",
  "adapted_at": "2026-06-22T10:00:00Z",
  "note": "可选说明"
}

PMO 记录 ACK 超时（默认 7 天）:
  - 超时未 ACK → PMO 告警（alert 级别）
  - 破坏性变更超时不 ACK → 暂停业务项目（escalation）
```

## 方向二：业务系统 → PMO（指标/事件/异常上报）

### 1. 三类上报主题

| 主题 | 消息类型 | QoS | 说明 |
|---|---|---|---|
| `biz.{id}.metric` | biz_data | 1 | 指标数据上报（周期） |
| `biz.{id}.exception` | alert | 0 | 异常告警（实时） |
| `biz.{id}.data` | biz_data | 2 | 业务数据上报（关键数据） |

### 2. 指标上报（biz.{id}.metric）

```
上报频率: 业务项目自行决定（建议每 5 分钟一次）
Payload:
{
  "biz_project_id": "1.2-finance",
  "phase": "P2-develop",
  "metrics": {
    "flow_latency": 120,        # 流程延迟（秒）
    "exception_rate": 0.02,     # 异常率
    "pass_rate": 0.95,           # 通过率
    "rollback_rate": 0.01,       # 回滚率
    "token_consumption": 3500    # Token 消耗
  },
  "timestamp": "2026-06-19T10:00:00Z",
  "period": "5min"
}
```

### 3. 异常上报（biz.{id}.exception）

```
触发条件: 业务项目内部异常（0.0.4 BizExceptionType 任意一种）
Payload:
{
  "biz_project_id": "1.2-finance",
  "exception_type": "VALIDATION_ERROR",
  "exception_code": "E2001",
  "message": "Schema validation failed for entity: position",
  "severity": "warning" | "critical",
  "context": {
    "phase": "P2-develop",
    "entity": "position",
    "field": "notional_amount"
  },
  "timestamp": "2026-06-19T10:00:00Z"
}
```

### 4. 业务数据上报（biz.{id}.data）

```
触发条件: 关键业务数据变更（业务项目自行定义哪些数据为"关键"）
Payload:
{
  "biz_project_id": "1.2-finance",
  "data_type": "position_snapshot",
  "data_key": "POS-20260619-001",
  "data_hash": "sha256:...",
  "payload": { ... },
  "timestamp": "2026-06-19T10:00:00Z"
}
```

```
│  │ (监听 E2/E3  │    │  pmo.schema.change  ←────────────┼───│── 轻量 Ping

          │                      ▲

   │  exception   │    │   exception      │
   │  schema.ack) │    │   schema.ack)    │
   └──────────────┘    └──────────────────┘
```

## 自检 D26 / D27

| 编号 | 自检项 | 验收点 |
|---|---|---|
| **D26** | Schema 变更通知自检 | ① E2/E3 变更触发事件；② 事件写入 sync-logs；③ 业务系统可订阅 `pmo.schema.change`；④ ACK 超时告警 |
| **D27** | 业务系统上报自检 | ① 业务系统可通过 broker 发送 metric；② 异常上报进 PMO 告警队列；③ 数据上报写入 PMO data-logs；④ 指标进 PMO 看板 |
