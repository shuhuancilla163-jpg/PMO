# m2.6-SOP 业务项目注册标准操作流程 (SOP)

**业务项目注册 SOP v0.2.0** — 基于 m2.6 + m2.1 + DEC-2026-0005，**含数据数仓与同步机制**

---

## 一、适用范围

本 SOP 适用于所有新业务项目接入 PMO 实例的注册操作。
已注册的业务项目变更（如版本升级、配置变更）走「变更流程」，不在本 SOP 范围内。

---

## 二、注册前准备

### 2.1 前置条件检查

业务项目发起注册前，必须确认以下前置条件已满足：

| # | 前置条件 | 检查方式 | 负责方 |
|---|---|---|---|
| 1 | 业务场景已确定 | 有明确业务场景描述 | 业务方 |
| 2 | Sponsor 已指定 | 有具名 Sponsor | 业务方 |
| 3 | 项目 ID 规划 | 确认 ID 不与现有项目冲突 | PMO |
| 4 | PMO 实例运行中 | PMO 服务状态正常 | PMO |

### 2.2 确定项目 ID

项目 ID 格式：`{序号}.{子序号}-{项目标识}`

| 示例 | 含义 |
|---|---|
| `1.1-pmo-self` | 序号 1.1，PMO 自建项目 |
| `1.2-finance` | 序号 1.2，量化金融项目 |
| `1.3-{your-biz}` | 序号 1.3，新业务项目 |

**规则**：
- 序号由 PMO 分配，业务方不可自定
- 项目标识使用英文字母+数字，不超过 20 字符
- ID 一经注册不可变更

## 三、注册操作（7 步）

```
[Step 0] PMO 预分配项目 ID
    ↓
[Step 1] 创建项目目录结构
    ↓
[Step 2] 填写 E1 业务项目元数据 (register.yaml)
    ↓
[Step 3] 填写 E2 业务数据 schema (data-schema.yaml)
    ↓
[Step 3.5] 配置数据同步机制 (data-sync.yaml)   ★ 新增：两仓分离 + 同步通道
    ↓
[Step 4] 填写 E3 业务术语表 (glossary.yaml)
    ↓
[Step 5] 签署 2 份契约
    ↓
[Step 6] PMO 验证并激活
```

### Step 0: PMO 预分配项目 ID

**执行方**：PMO 层

**操作**：

1. 检查现有项目 ID 列表，确认下一个可用序号
2. 生成项目 ID：`{序号}.{子序号}-{项目标识}`
3. 向业务方返回分配结果

**输出**：

```yaml
allocation:
  project_id: "1.3-my-project"
  allocated_at: "2026-06-19T18:00:00Z"
  allocated_by: "PMO-L1"
```

### 数据数仓分离原则（两仓独立）

**核心原则**：PMO 数仓与业务项目数仓完全隔离，业务系统不直接写入 PMO 数仓。

#### 两仓定位

| 仓库 | 存储内容 | 管理者 | 业务系统访问 |
|---|---|---|---|
| **PMO 数仓** | PMO 治理数据：E1/E2/E3 元数据、决策日志、状态报告、指标元数据 | PMO | 仅 PMO 可写；业务系统只读元数据 Schema |
| **业务项目数仓** | 业务数据：业务实体原始数据、业务流程数据、业务指标原始值、业务日志 | 业务系统 | 业务系统全权管理；PMO 只读业务Schema（E2） |

#### PMO 数仓目录结构

```
PMO/
└── data-warehouse/                  # PMO 数仓（PMO 全权管理）
    ├── biz-meta/                     # E1/E2/E3 元数据（业务项目注册时写入）
    │   ├── 1.1-pmo-self.json
    │   ├── 1.1-pmo-self-schema.json
    │   ├── 1.1-pmo-self-glossary.json
    │   └── ...
    ├── decisions/                    # 决策日志（不可变）
    │   └── active/
    ├── states/                      # 业务项目状态快照
    │   └── biz-project-states/
    ├── metrics/                     # 指标元数据（不含业务原始值）
    │   └── meta/
    ├── sync-logs/                   # 数据同步操作日志
    │   └── sync-events/
    └── archive/                     # 归档（按 4 级归档）
        ├── biz-projects/
        ├── tasks/
        └── decisions/
```

#### 业务项目数仓（业务系统自建）

```
业务系统/
└── data-warehouse/                  # 业务项目数仓（业务系统全权管理）
    ├── raw/                         # 原始业务数据（业务系统写入）
    ├── processed/                   # 加工后业务数据
    ├── metrics/                     # 业务指标原始值（业务系统采集）
    │   └── {project-id}/
    │       ├── flow_latency/
    │       ├── exception_rate/
    │       └── ...
    └── logs/                       # 业务日志
```

**PMO 绝对不写入业务项目数仓；业务系统绝对不直接写入 PMO 数仓。**

### Step 2: 填写 E1 业务项目元数据 (register.yaml)

**执行方**：业务方

**必填字段**（7 大块）：

```yaml
# 1. 业务项目基本信息
biz_project:
  id: "1.3-my-project"          # Step 0 分配的项目 ID
  name: "我的业务项目"             # 项目中文名
  type: "my-biz-type"            # 业务类型标识
  version: "0.1.0"               # 初始版本
  sponsor: "Sponsor"             # Sponsor 名称
  registered_at: "2026-06-19"    # 注册日期
  phase: "P0-init"               # 当前阶段

# 2. PMO 监管配置 (必填)
pmo_supervision:
  enabled: true
  state: "active"
  quota_4d:                        # 4 维配额 (必填)
    token: "100000/day"           # 日 Token 上限
    time: "24/7"                  # 运行时间
    storage: "10GB"               # 存储配额
    concurrency: "5"              # 并发上限
  isolation_3d:                   # 3 维隔离 (必填)
    data: "full"                  # 数据隔离级别
    config: "independent"         # 配置隔离
    state: "independent"          # 状态隔离

# 3. 2 类 agent 分类 (必填, DEC-2026-0003)
agent_categories:
  eng_roles:
    location: "eng-roles/"
    pmo_supervised: true          # 研发 agent 由 PMO 监管
  biz_agents:
    location: "biz-agents/"
    pmo_supervised: false         # 业务 agent 业务自管

# 4. 2 层合规声明 (必填, 0.0.11)
compliance_2_layers:
  layer_1_overall: true          # 业务项目整体合规
  layer_2_eng_5_stages: true      # 研发 5 阶段合规

# 5. m2.6 7 项声明 (必填)
m2_6_7_items:
  registration: true
  state_machine: true
  quota_4d: true
  archive_4_levels: true
  isolation_3d: true
  alerting_3_levels: true
  checklist_6_plus_3: true

# 6. 治理规范引用
governance_refs:
  - "immutable/0-governance/0.0.1-five-values.md"
  - "immutable/0-governance/0.0.2-engineering.md"
  - "immutable/0-governance/0.0.10-1spec-Nprojects.md"
  - "immutable/0-governance/0.0.11-two-layer-compliance.md"

# 7. 业务指标 (基础 5 项 + 自定义)
metrics:
  business_basic_5:
    - flow_latency
    - exception_rate
    - pass_rate
    - rollback_rate
    - token_consumption
  custom: []
```

**检查点**：

- [ ] `biz_project.id` 与 Step 0 分配的 ID 一致
- [ ] `quota_4d` 4 维全部填写
- [ ] `isolation_3d` 3 维全部填写
- [ ] `compliance_2_layers` 两层均为 true
- [ ] `m2_6_7_items` 7 项均为 true
- [ ] `sponsor` 已指定（非空）

### Step 3: 填写 E2 业务数据 schema (data-schema.yaml)

**执行方**：业务方

**必填内容**：定义业务数据实体和业务事件

```yaml
# E2 业务数据 Schema (m2.1, DEC-2026-0005)
schema_version: "0.1.0"
project_id: "1.3-my-project"
defined_by: "biz_project"
stored_by: "pmo_instance"

entities:
  - name: "task"                  # 实体名（英文字母）
    description: "任务实体"
    primary_key: "task_id"
    fields:
      - name: "task_id"          # 字段名
        type: "string"
        required: true
        description: "任务唯一标识"
      - name: "title"
        type: "string"
        required: true
      - name: "status"
        type: "enum"
        values: ["pending", "in_progress", "completed", "failed"]
        required: true
      - name: "priority"
        type: "enum"
        values: ["P0", "P1", "P2", "P3"]
        required: true
      - name: "created_at"
        type: "datetime"
        required: true

events:
  - name: "task_started"
    description: "任务开始事件"
    fields:
      - name: "task_id"
        type: "string"
        required: true
      - name: "started_at"
        type: "datetime"
        required: true

  - name: "task_completed"
    description: "任务完成事件"
    fields:
      - name: "task_id"
        type: "string"
        required: true
      - name: "completed_at"
        type: "datetime"
        required: true
```

**PMO 验证规则**（自动检查）：

| 规则 | 描述 |
|---|---|
| E2-R1 | 实体名唯一，不与已有项目实体名冲突 |
| E2-R2 | primary_key 字段在 fields 内存在 |
| E2-R3 | enum.values 必填，非空 |
| E2-R4 | required=true 的字段必须有 |
| E2-R5 | 事件 fields 中 required=true 的字段必须有 |

### Step 3.5: 配置数据同步机制 (data-sync.yaml)

**执行方**：业务方（业务系统侧）

**前提**：业务系统必须在正式接入 PMO 前完成数据同步机制的配置。PMO 不管理业务项目数仓，业务系统负责建立和维护同步通道。

#### 两仓分离原则

**核心原则**：PMO 数仓与业务项目数仓完全隔离，业务系统不直接写入 PMO 数仓。

| 仓库 | 存储内容 | 管理者 | 访问规则 |
|---|---|---|---|
| **PMO 数仓** | PMO 治理数据：E1/E2/E3 元数据、决策日志、状态快照、指标元数据 | PMO 全权管理 | 业务系统只读 Schema；不直接写入 |
| **业务项目数仓** | 业务数据：业务实体原始数据、业务流程数据、业务指标原始值、业务日志 | 业务系统全权管理 | PMO 只读 E2 Schema；不读取原始数据 |

**PMO 绝对不写入业务项目数仓；业务系统绝对不直接写入 PMO 数仓。**

#### 同步方向

| 方向 | 路径 | 数据内容 |
|---|---|---|
| **业务 → PMO** (push) | 业务系统主动上报 | 业务指标元数据（5 项基础 + 自定义）、业务事件（E2.events） |
| **PMO → 业务** (pull) | PMO 主动通知 | Schema 变更通知、E2/E3 版本更新、业务项目状态变更 |

#### data-sync.yaml 填写模板

```yaml
# 数据同步配置 (m2.6-SOP, 业务系统侧维护)
sync_version: "0.1.0"
project_id: "1.3-my-project"

# ===== 同步目标声明 =====
sync_targets:
  # 业务 → PMO 上报通道
  push:
    enabled: true
    endpoint: "https://{pmo-host}/api/v1/biz/{project-id}/report"
    protocol: "HTTPS-REST"         # 或 "MCP" / "Webhook"
    auth:
      type: "API-Key"
      key_ref: "env:PMO_API_KEY"   # 从环境变量读取，不硬编码

  # PMO → 业务通知通道
  pull:
    enabled: true
    endpoint: "https://{pmo-host}/api/v1/biz/{project-id}/subscribe"
    protocol: "Webhook"
    auth:
      type: "HMAC-Signature"       # PMO 用 HMAC 签名验证通知真实性
      key_ref: "env:PMO_WEBHOOK_SECRET"

# ===== 同步数据流声明 =====
data_streams:
  # 流 1: 业务指标上报
  - stream_id: "metrics-flow"
    direction: "push"              # 业务 → PMO
    source:
      type: "biz-metrics"
      path: "biz-data-warehouse/metrics/{project-id}/"
      format: "JSON-Lines"
    destination:
      type: "pmo-metrics-meta"
      path: "data-warehouse/metrics/{project-id}/"
    frequency: "hourly"            # 同步频率: real-time / hourly / daily
    schema_ref: "E2"
    filter:
      include_metrics:
        - "flow_latency"
        - "exception_rate"
        - "pass_rate"
        - "rollback_rate"
        - "token_consumption"

  # 流 2: 业务事件上报
  - stream_id: "events-flow"
    direction: "push"
    source:
      type: "biz-events"
      path: "biz-data-warehouse/events/{project-id}/"
      format: "JSON"
    destination:
      type: "pmo-bus"
      path: "data-warehouse/sync-logs/events/"
    frequency: "real-time"         # 业务事件实时上报
    schema_ref: "E2-events"
    filter:
      include_events:
        - "task_started"
        - "task_completed"
        - "decision_made"
        - "metric_recorded"

  # 流 3: PMO Schema 变更通知
  - stream_id: "schema-change-notify"
    direction: "pull"
    source:
      type: "pmo-meta"
      path: "config/biz-meta/{project-id}-schema.json"
    destination:
      type: "biz-data-warehouse"
      path: "biz-data-warehouse/schema-inbox/"
    frequency: "on-change"
    action_on_receive: "ALERT"     # 收到后: ALERT / AUTO_UPDATE / MANUAL_REVIEW

# ===== Schema 变更处理规则 =====
schema_change_policy:
  # E2 (data-schema) 变更处理
  e2_change:
    on_entity_add: "ALERT"         # 新增实体: 告警确认
    on_entity_remove: "REJECT"      # 删除实体: 禁止
    on_field_add: "ALERT"           # 新增字段: 告警确认
    on_field_remove: "REJECT"       # 删除字段: 禁止
    on_type_change: "ALERT"        # 类型变更: 告警

  # E3 (glossary) 变更处理
  e3_change:
    on_term_add: "AUTO_ACCEPT"     # 新增术语: 自动接受
    on_term_deprecate: "ALERT"     # 废弃术语: 告警
    on_role_add: "ALERT"           # 新增角色: 告警

  # 禁止行为（必须全部声明）
  prohibited:
    - "业务系统直接写入 PMO 数仓"
    - "业务系统删除 PMO 元数据"
    - "PMO 读取业务项目数仓原始数据"
    - "绕过同步机制的直接数据写入"

# ===== 同步状态报告 =====
sync_status:
  last_push_at: null
  last_pull_at: null
  last_error_at: null
  error_count: 0
```

#### PMO 验证规则（D19 数据同步自检）

| 规则 | 描述 | 失败处理 |
|---|---|---|
| D19-R1 | data-sync.yaml 存在且格式正确 | 打回业务方补填 |
| D19-R2 | push 通道至少配置 1 个（metrics 或 events） | 打回补填 |
| D19-R3 | auth 配置引用环境变量，不硬编码密钥 | 打回修正 |
| D19-R4 | schema_change_policy 中 on_entity_remove = REJECT | 打回修正 |
| D19-R5 | prohibited 列表包含全部 4 项禁止行为 | 打回补填 |
| D19-R6 | 同步频率至少声明一个（real-time / hourly / daily） | 打回补填 |

#### 业务系统接入检查清单（数据同步部分）

业务系统接入 PMO 前，必须完成以下所有检查项：

- [ ] 业务项目数仓已建立（`biz-data-warehouse/` 目录存在）
- [ ] push 通道已配置（业务 → PMO 上报）
- [ ] pull 通道已配置（PMO → 业务通知）
- [ ] auth 机制已配置（密钥引用环境变量，不硬编码）
- [ ] 同步数据流至少声明 1 条
- [ ] Schema 变更策略已配置（禁止删除字段/实体）
- [ ] 禁止行为已全部声明
- [ ] 已完成首次同步测试（test-sync 命令）

### Step 4: 填写 E3 业务术语表 (glossary.yaml)

**执行方**：业务方

**必填内容**：定义业务术语和业务 agent 角色

```yaml
# E3 业务术语表 (m2.1, DEC-2026-0005)
glossary_version: "0.1.0"
project_id: "1.3-my-project"
created_by: "biz_project"

# 业务术语定义
terms:
  - term: "task"                  # 术语名
    category: "entity"           # 类别: entity/state/metric/action/role
    definition: "业务中可独立执行的工作单元"
    example: "行情数据采集是一个 task"
    english: "task"              # 英文对照

  - term: "flow"
    category: "action"
    definition: "业务数据处理流程"
    example: "行情数据从采集到存储的完整流程"
    english: "flow"

# 业务 agent 角色定义 (2 类之一: biz_agent)
biz_agent_roles:
  - role: "MyProject-Operator"
    category: "role"
    description: "负责业务流程执行的 agent"
    capabilities:
      - "执行流程"
      - "处理异常"
    layer: "L2"                  # 业务 agent 层级

  - role: "MyProject-Reporter"
    category: "role"
    description: "负责数据上报的 agent"
    capabilities:
      - "采集指标"
      - "生成报告"
    layer: "L2"
```

**PMO 验证规则**：

| 规则 | 描述 |
|---|---|
| E3-R1 | 业务 agent role 不得与 5 阶段研发 role 重名（Requirement-Engineer / Development-Engineer / Test-Engineer / Operations-Engineer / Evaluation-Engineer） |
| E3-R2 | term.category 必填，且为 entity/state/metric/action/role 之一 |
| E3-R3 | biz_agent_roles 每条必须有 description |

### Step 5: 签署 2 份契约

**执行方**：业务方 + PMO

**契约 1：业务项目整体契约**

路径：`PMO/immutable/2-biz-specs/contract-{project-id}-overall.md`

内容包含：
- 业务项目基本信息（E1 摘要）
- PMO 监管条款（m2.6 7 项承诺）
- 2 层合规声明
- 退出/归档条款
- 签署方 + 签署时间

**契约 2：业务项目内研发 5 阶段契约**

路径：`PMO/immutable/2-biz-specs/contract-{project-id}-eng-5-stages.md`

内容包含：
- 研发 5 阶段范围声明
- PMO 7 项合规承诺
- 各阶段角色定义
- 阶段门控规则
- 签署方 + 签署时间

**签署检查点**：

- [ ] 2 份契约文件已创建
- [ ] 业务方 Sponsor 已确认
- [ ] PMO 已盖章（签名 + Hash）
- [ ] 签署时间已记录

### Step 6: PMO 验证并激活

**执行方**：PMO 层（L1 PMO-Main）

**验证清单（6 项）**：

| # | 验证项 | 验证内容 | 不通过处理 |
|---|---|---|---|
| V1 | E1 完整性 | register.yaml 7 大块必填字段全填 | 打回业务方补填 |
| V2 | E2 格式正确 | data-schema.yaml 通过 E2-R1~R5 规则 | 打回业务方修正 |
| V3 | E3 边界合规 | glossary.yaml 通过 E3-R1~R3 规则 | 打回业务方修正 |
| V4 | 2 份契约已签 | 契约文件存在 + Hash 有效 | 打回业务方签署 |
| V5 | 目录结构完整 | Step 1 的目录结构全部创建 | 打回业务方补建 |
| V6 | ID 不冲突 | 项目 ID 不与现有项目重复 | 重新分配 ID |
| **V7** | **data-sync.yaml 完整** | **D19-R1~R6 全部通过** | **打回业务方补填** |

**激活操作**：

验证全部通过后，PMO 执行以下激活操作：

1. **E1 写入 PMO 配置**：`config/biz-meta/{project-id}.json`
2. **E2 写入 PMO 配置**：`config/biz-meta/{project-id}-schema.json`
3. **E3 写入 PMO 配置**：`config/biz-meta/{project-id}-glossary.json`
4. **项目状态设为 active**：`tasks/state/{project-id}.json`
5. **初始化研发 5 阶段模板**：`biz-projects/{project-id}/eng-roles/`
6. **向 Sponsor 发送注册完成通知**
7. **记录决策日志**：`DEC-{year}-{seq}` → `decisions/active/`

**激活输出**：

```yaml
activation:
  project_id: "1.3-my-project"
  status: "active"
  activated_at: "2026-06-19T18:30:00Z"
  activated_by: "PMO-L1"
  pmo_meta_stored:
    - "config/biz-meta/1.3-my-project.json"              # E1
    - "config/biz-meta/1.3-my-project-schema.json"        # E2
    - "config/biz-meta/1.3-my-project-glossary.json"      # E3
    - "config/biz-meta/1.3-my-project-sync.json"          # data-sync.yaml (PMO 只存配置，不存业务数据)
  state_file: "tasks/state/1.3-my-project.json"
  decision_id: "DEC-2026-XXXX"
```

## 五、注册后 5 步接入流程

注册完成后，业务项目按以下 5 步正式接入 PMO：

```
Step 1 (已完成): 业务项目注册 ← 注册即完成
    ↓
Step 2: 研发 5 阶段声明 (eng-roles/ 填写)
    ↓
Step 3: 业务配置 (业务方自管)
    ↓
Step 4: 接入 PMO 消息机制 (messaging.yaml 填写)
    ↓
Step 5: 接入 PMO 监管 (checklist.md + 指标上报)
```

**数据同步是注册前置条件**：Step 3.5 完成后，业务系统即可开始向 PMO 数仓上报数据，不需等待以上 5 步完成。

**参考**：`docs/biz-project-onboarding-5-steps.md`

## 六、常见问题

| # | 问题 | 解答 |
|---|---|---|
| F1 | register.yaml 的 quota_4d 怎么填？ | 按业务实际需求估算，PMO 后续可调整。日 Token 参考同类项目：PMO 自建 10M/day，量化金融 100K/day |
| F2 | E2 的实体很多怎么办？ | 按实际业务需求定义，PMO 不限制数量，但 primary_key 和 required 字段必须完整 |
| F3 | 业务 agent role 和研发 agent role 有什么区别？ | 研发 agent (eng-roles/) 由 PMO 预设监管；业务 agent (biz-agents/) 由业务方自管，两者不能重名 |
| F4 | 注册后发现 E1/E2/E3 有错误怎么办？ | 走变更流程，更新对应文件，PMO 重新验证，重新签署契约 |
| F5 | 可以跳过 Step 3 和 Step 4 吗？ | E2 和 E3 必填（m2.1 强制要求），但具体内容可先填最小集，后续迭代补充 |
| F6 | 业务项目数仓和 PMO 数仓必须物理隔离吗？ | 是。两仓在概念、物理和管理权限上完全隔离。业务系统不直接写 PMO 数仓；PMO 不读取业务原始数据 |
| F7 | PMO 读取业务项目数仓吗？ | 不读取。PMO 只读取业务系统上报的指标元数据（通过 data-sync push 通道）；不读取业务项目数仓的原始数据 |
| F8 | 数据同步失败怎么办？ | 按 m0.4 恢复策略处理（RETRY → FALLBACK → ESCALATE）；同步错误计入 sync_status.error_count；达到阈值触发 L2 告警 |

## 七、关键决策

- **m2.6-SOP v0.2.0**: 业务项目注册 SOP，含数据数仓与同步机制 (2026-06-19)
- **7 步注册**: ID 分配 → 目录创建 → E1 → E2 → **data-sync → E3** → 契约签署 → PMO 激活
- **两仓绝对分离**: PMO 数仓（PMO 全权）≠ 业务项目数仓（业务系统全权）
- **同步方向固定**: 业务 → PMO（push）上报指标/事件；PMO → 业务（pull）通知 Schema 变更
- **D19 数据同步自检**: 6 项验证规则，data-sync.yaml 不完整不能激活
- **E1/E2/E3 必填**: m2.1 强制要求（DEC-2026-0005）
- **2 份契约必签**: 业务项目整体 + 研发 5 阶段
- **PMO 验证激活**: 7 项验证全部通过后才能激活
