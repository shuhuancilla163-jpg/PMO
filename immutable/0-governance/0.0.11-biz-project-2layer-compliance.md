# 0.0.11 业务项目 2 层合规元规则 (DEC-2026-0002)

**业务项目 = 1 个单位, 但 PMO 监管 2 层合规** (Sponsor 平等讨论后定)

## 核心洞察

**业务项目 1 个单位, 内部分 2 组**:
- **业务组 (业务项目自管)**: 业务 agent / 业务流程 / 业务异常 / 业务指标 → 业务项目自维护
- **研发组 (PMO 监管)**: 5 阶段研发角色 (需求/研发/测试/运维/评估) → 业务项目接入 PMO 监管

## 业务项目 2 层合规

### 第 1 层: 业务项目整体 (PMO 监管)

**业务项目作为 1 个单位, 符合 PMO m2.6 7 项**:
1. **业务项目注册** (m2.6 业务注册)
2. **业务项目状态机** (active/paused/blocked/completed/archived)
3. **业务项目配额 4 维** (Token/时间/存储/并发)
4. **业务项目归档 4 层面** (数据/文档/项目/资源)
5. **业务项目隔离 3 维** (数据/配置/状态)
6. **业务项目告警 3 层** (业务自给/PMO/Sponsor)
7. **业务项目 checklist 6+3 项** (基础 6 + Sponsor/复盘/规则)

**监管**: L1 PMO-Main (维度 1 采集)

### 第 2 层: 业务项目内研发 5 阶段 (PMO 监管)

**业务项目内的研发, 符合 PMO 7 项**:
1. **阶段门控** (m1.3)
2. **决策日志** (m1.1)
3. **不可变文档** (m0.1)
4. **异常拦截** (m0.4)
5. **指标可贯彻** (m1.2)
6. **3 层告警** (m0.4)
7. **Sponsor 报告** (m1.5)

**5 阶段研发角色** (PMO 给初始模板, 业务项目实施/调整 — DEC-2026-0003):
1. **Requirement-Engineer** (需求) — 可选
2. **Development-Engineer** (研发) — 必选
3. **Test-Engineer** (测试) — 必选
4. **Operations-Engineer** (运维) — 必选
5. **Evaluation-Engineer** (评估) — 可选

**关键澄清 (DEC-2026-0003)**:
- 5 阶段 agent **物理位置在业务项目内** (`biz-projects/<biz_project_id>/eng-roles/`), 由业务项目实施
- **PMO 不实施 5 阶段 agent**, 只提供初始模板 (`PMO/templates/eng-roles/`) + 输入输出规范 (0.0.13) + 7 项合规监督
- 业务项目**可调** 5 阶段 agent (基于 PMO 模板 + 结合业务), 但必须符合 PMO 7 项
- 业务项目接入时从 PMO 模板目录复制到 `eng-roles/`, 然后按业务调整

**监管**: L2 Engineer-Agent (维度 2 采业务项目 5 阶段上报数据, 不实施)

## 业务项目内 2 类 agent 边界 (DEC-2026-0003)

**业务项目内 agent 严格分 2 类, 边界清晰**:

### 1. 5 阶段研发 agent (PMO 监管)
- 物理位置: `biz-projects/<biz_project_id>/eng-roles/`
- 实施: **业务项目自己实施** (从 PMO 模板复制 + 调整)
- PMO 职责: 给初始模板 + 输入输出规范 + 7 项合规监督
- PMO **不**实施 5 阶段 agent
- 阶段: requirement / development / test / operations / evaluation
- 元规则: 0.0.13 (5 阶段输入输出规范)

### 2. 业务需求运营 agent (业务自管)
- 物理位置: `biz-projects/<biz_project_id>/biz-agents/`
- 实施: **业务项目完全自定** (按业务场景配置, 不固定 8 个)
- PMO 职责: **不预设不干预内容** (业务项目自管)
- 业务项目自定: 业务场景 / 业务 agent 名字 / 业务 agent 数量 / 业务流 / 业务异常 / 业务指标
- 元规则: 0.0.14 (业务 agent 自定原则)

**PMO 监督边界**:
- 5 阶段 agent: PMO 看上报数据是否合规, 不看具体实现代码
- 业务 agent: PMO **不监督内容** (业务自管, PMO 只看业务项目整体 m2.6 7 项合规)

## 业务项目内业务自管 (PMO 不干预)

**业务项目自维护** (PMO 不干预业务内容):
- **业务场景** (业务项目自选, 后期定, 不预设模板)
- **业务 agent** (业务项目按业务场景配置, 不固定 8 个)
- **业务流程** (业务项目自维护业务流模板)
- **业务异常定义** (业务项目自定义具体业务异常)
- **业务指标全量** (业务项目自维护业务全量数据)
- **业务数据存储** (业务项目自存, PMO 不存)

## 业务项目接入 PMO 5 步流程

1. **业务项目注册** (m2.6) — 业务项目填 `register.yaml` + 签 2 份契约
2. **业务项目内研发 5 阶段声明** (m2.3 + m2.6) — 业务项目在 `eng-roles/` 声明
3. **业务项目内业务配置** (业务项目自管) — 业务场景/业务 agent/业务流程/业务异常/业务指标
4. **业务项目接入 PMO 消息机制** (m2.5) — `messaging.yaml` 声明订阅/发布
5. **业务项目接入 PMO 监管** (m1.3 + m0.4) — `reports/` 上报关键指标

## 关键

- **业务项目 = 1 个单位**, 不是 2 个单位
- **业务项目 2 层合规**: 业务项目整体 + 业务项目内研发 5 阶段
- **业务项目自管业务**: 业务 agent / 业务流程 / 业务异常 / 业务指标
- **PMO 监管**: 注册/状态/配额/隔离 + 研发 5 阶段符合 PMO 7 项
- **业务项目接入 5 步流程**: 业务项目接入 PMO 的标准流程
