# 0.0.12 3 维度架构元规则 (DEC-2026-0002)

**业务项目 = 1 个单位, 但 PMO 监管 3 维度** (Sponsor 平等讨论后定)

## 核心洞察

**不同维度考核指标不同 + 数据源不同, PMO 角色必须严格按 3 维度分离** (Sponsor 关键点出)。

## 3 维度架构

| 维度 | 数据源 | 谁采 (PMO 角色) | 谁存 | 谁用 |
|---|---|---|---|---|
| **维度 1: 业务项目整体** | 业务项目注册/状态/配额/归档/隔离 | **L1 PMO-Main** | PMO 存 | PMO 监管 + Sponsor 看 |
| **维度 2: 业务项目内研发 5 阶段** | 研发 5 阶段数据 (需求/研发/测试/运维/评估) | **L2 PMO-Engineer-Agent** 按阶段采 | PMO 存 (上报) + 业务项目存 (全量) | PMO 监管 + 业务项目看 |
| **维度 3: 业务项目内业务** | 业务项目按 PMO 规范上报的关键指标 | **L2 PMO-Monitor-Agent** 采上报数据 | 业务项目存业务全量 + PMO 存上报数据 | 业务项目用业务数据, PMO 监管上报合规 |

## 关键洞察 (Sponsor 推动)

### 1. 业务数据全量 = 业务项目自存 (PMO 不存)
- 业务项目 = "上市公司" (自己业务数据自己存)
- 业务数据全量 = "公司全部数据" (PMO 不存)

### 2. 业务数据上报 = 业务项目按 PMO 规范上报关键指标
- 业务项目上报 = "上市公司按 SEC 规范上报季报" (关键指标按规范上报)
- 上报数据 = "季报" (PMO 存上报数据)

### 3. PMO 监管 = 看上报数据是否合规, 不看业务具体内容
- PMO = "SEC 监管机构" (看上报数据, 不知道公司具体业务)
- PMO 监管 = "看季报是否合规" (不看具体业务)

## 3 维度采集角色严格分离

| 角色 | 采什么 | 数据源 |
|---|---|---|
| **L1 PMO-Main** | 维度 1 (业务项目整体) | register.yaml + state + quota + archive + isolation |
| **L2 PMO-Engineer-Agent** | 维度 2 (研发 5 阶段) | eng-roles/ + 5 阶段产出 + 决策日志 + 不可变文档 |
| **L2 PMO-Monitor-Agent** | 维度 3 (业务项目上报) | 业务项目按 PMO 规范上报的关键指标 |

**关键**:
- 3 维度采集**不重叠**, 严格分离
- 1 个角色只采 1 个维度 (避免混乱)
- 1 个维度只被 1 个角色采 (避免重复)

## 3 维度考核 (L2 PMO-Assessor-Agent)

| 考核对象 | 考核内容 | 考核指标 |
|---|---|---|
| **业务项目整体** | 业务项目是否符合 PMO m2.6 7 项 | biz_project_compliance_rate, biz_project_quota_usage, biz_project_state_legality, biz_project_archive_completeness, biz_project_isolation_violation, biz_project_alert_count |
| **业务项目内研发 5 阶段** | 研发 5 阶段是否符合 PMO 7 项 | 按 5 阶段分别: requirement_phase_compliance, development_phase_compliance, test_phase_compliance, operations_phase_compliance, evaluation_phase_compliance |
| **业务项目内业务** | 业务项目是否按 PMO 规范上报 | biz_metrics_report_compliance, biz_metrics_report_timeliness, biz_metrics_report_completeness |

## 3 维度监控 (3 个采集角色同时是监控)

| 监控对象 | 监控方 | 监控内容 |
|---|---|---|
| **业务项目状态** | L1 PMO-Main | active/paused/blocked/completed/archived 状态变化 |
| **业务指标** | L2 PMO-Monitor-Agent | 5 项基础 (流耗时/异常率/通过率/回滚率/Token) + 业务项目自定义 |
| **业务异常** | L2 PMO-Monitor-Agent | 3 层异常拦截 (业务/PMO/规范) 异常次数/严重性/拦截率 |
| **业务资源** | L1 PMO-Main | 4 维配额 (Token/时间/存储/并发) 使用率 |
| **项目间消息** | L2 PMO-Message-Broker-Agent | 数量/成功率/延迟/失败重试 |

## 3 层上报机制

| 层 | 谁采 | 数据内容 | 谁存 |
|---|---|---|---|
| **L1 业务项目** | 业务项目自采 (业务流/业务 agent/业务指标) | 业务全量数据 | 业务项目自己 |
| **L2 业务项目上报 PMO** | 业务项目按 PMO 规范**上报关键指标** | 业务关键指标 (5 项基础 + 业务项目自定义) | PMO 存上报数据 |
| **L3 PMO 监管** | PMO 看上报数据, 监管是否合规 | 上报数据 + 监管结论 | PMO 存监管结论 |

## 8 PMO 角色 (按 3 维度严格分离)

| # | 层级 | 角色 | 职责 | 数据维度 |
|---|---|---|---|---|
| 1 | L0 | Sponsor | 监督权 (人机, 看指标看板, 不执行) | 3 维度综合 |
| 2 | L1 | PMO-Main | 行政权 (1 实例, 管 N 项目) | **维度 1** (业务项目整体) |
| 3 | L2 | PMO-Plan-Agent | 司法权 (计划/治理) | 维度 1 (项目治理) |
| 4 | L2 | PMO-Engineer-Agent | 司法权 (工程) | **维度 2** (研发 5 阶段) |
| 5 | L2 | PMO-Monitor-Agent | 司法权 (监控) | **维度 3** (业务项目上报) |
| 6 | L2 | PMO-Reviewer-Agent | 司法权 (审查/审计) | 3 维度综合审查 |
| 7 | L2 | PMO-Assessor-Agent | 司法权 (考核) | **3 维度分别考核** |
| 8 | L2 | PMO-Message-Broker-Agent | 司法权 (消息) | **项目间消息** (业务↔业务 经 PMO 中介) |

## 业务项目间消息 (L2 PMO-Message-Broker-Agent)

**业务项目↔业务项目 (1.2↔1.3) 消息经 PMO 实例中介**:

| 消息类型 | 含义 |
|---|---|
| request/response | 同步 |
| notification/alert/escalation | 异步 |
| biz_event | 业务项目状态变化/业务指标超阈值/业务异常 |
| biz_data | 业务结果/业务报告 |

**消息主题**:
- `biz.{biz_project_id}.state`
- `biz.{biz_project_id}.metric`
- `biz.{biz_project_id}.exception`
- `biz.{biz_project_id}.data`
- `inter.biz.{biz_project_id_a}.to.{biz_project_id_b}`

**消息协议**:
- 协议: PMO 通信协议 (m0.2 protocol.py 扩展)
- 格式: JSON
- QoS: at-least-once
- 可监控 + 可审计

## 关键

- **不同维度考核指标不同, 数据源不同, PMO 角色严格按 3 维度分离**
- **业务数据全量 = 业务项目自存, PMO 不存**
- **业务数据上报 = 业务项目按 PMO 规范上报关键指标**
- **PMO 监管 = 看上报数据是否合规, 不看业务具体内容**
- **3 维度采集不重叠**: PMO-Main / Engineer-Agent / Monitor-Agent
- **3 维度考核**: Assessor-Agent
- **项目间消息经 PMO 中介**: Message-Broker-Agent

## 关键决策

- **DEC-2026-0002**: 14 块关键调整 (2026-06-18)
- **3 维度架构**: 业务项目整体 / 研发 5 阶段 / 业务项目内业务
- **8 PMO 角色**: 按 3 维度严格分离
- **3 层上报机制**: 业务自采 + 业务上报 + PMO 存上报
- **3 维度考核**: Assessor-Agent 按 3 维度分别考核
- **项目间消息经 PMO 中介**: Message-Broker-Agent

**不可变 (Sponsor 定的 3 维度架构原则)**
