# 0.0.15 项目间消息流通 (经 PMO 中介, DEC-2026-0004)

**元规则: 业务项目↔业务项目消息经 PMO 实例中介 (m1.6, DEC-2026-0004)**

## 1. 原则

业务项目间**不直接通信**, 全部经 PMO 实例的 Message-Broker-Agent 中介。

- **业务项目** = 分公司
- **PMO 实例** = 总公司邮件系统
- **项目间消息** = 分公司间的邮件, 都走邮件系统

## 2. 关键设计

| 维度 | 规范 |
|---|---|
| **路由** | 业务项目 A → PMO-Message-Broker → 业务项目 B (单向中介) |
| **避免耦合** | 业务项目不知道对方地址, 只订阅主题 |
| **可监控** | PMO 实例管所有消息数量/成功率/延迟/重试 |
| **可审计** | 消息日志 (append-only, 决策日志规范) |
| **可重试** | QoS = at-least-once, 失败重试由 broker 管 |
| **可扩展** | 加新业务项目时, PMO 实例自动接入消息机制 |

## 3. 消息模型 (强制)

每条消息必须包含以下字段:

```yaml
- msg_id: 8 字符 UUID (m0.2 protocol.py 已有)
- msg_type: 6 类必选其一
  - request        # 请求 (业务项目 A → B, 等响应)
  - response       # 响应 (B → A)
  - notification   # 通知 (单向, 异步)
  - alert          # 告警 (单向, 异步)
  - escalation     # 升级 (单向, 异步, 上行)
  - biz_event      # 业务事件 (业务项目状态变化/业务指标超阈值/业务异常)
  - biz_data       # 业务数据 (业务结果/业务报告)
- from: 发送方 (项目 ID 或 agent 名称)
- to: 接收方 (项目 ID 或 agent 名称, 业务项目间消息经 broker)
- topic: 消息主题
- qos: 0=fire-and-forget / 1=at-least-once (默认) / 2=exactly-once
- timestamp: ISO 8601
- layer: L0/L1/L2/biz
- content: 业务负载
- correlation_id: 关联 ID (request↔response, biz_event↔biz_data)
- retry_count: 重试次数 (broker 自动加)
- status: pending / delivered / failed / acked
```

## 4. 消息主题 (强制命名规范)

```
biz.{biz_project_id}.state           # 业务项目状态变化
biz.{biz_project_id}.metric          # 业务项目指标上报
biz.{biz_project_id}.exception       # 业务项目异常
biz.{biz_project_id}.data            # 业务项目数据
inter.biz.{a}.to.{b}                # 业务项目间定向消息
pmo.{layer}.{action}                # PMO 内部消息 (L0/L1/L2)
```

**业务项目只能订阅自己有权的主题** (由 PMO 校验)。

## 5. 协议 (强制)

- **传输**: JSON
- **QoS 默认**: at-least-once (1)
- **可选**: fire-and-forget (0) / exactly-once (2)
- **格式校验**: broker 收到消息必须校验必填字段, 缺字段拒收
- **版本**: protocol_version 字段 (当前 "1.0")

## 6. 监控 (L2 PMO-Message-Broker-Agent)

| 监控项 | 含义 | 收集方 |
|---|---|---|
| message_total_sent | 累计发布 | broker 自动 |
| message_total_delivered | 累计投递 | broker 自动 |
| message_total_failed | 累计失败 | broker 自动 |
| message_total_retried | 累计重试 | broker 自动 |
| message_avg_latency_ms | 平均投递延迟 | broker 自动 |
| message_success_rate | 投递成功率 (delivered / (delivered+failed)) | broker 计算 |
| subscriptions_count | 订阅数 | broker 实时 |
| topics_count | 主题数 | broker 实时 |
| pending_count | 待投递数 | broker 实时 |
| audit_log_count | 审计日志条数 | broker 自动 |

## 7. 审计 (强制, append-only)

每条消息都写审计日志, 字段:

```yaml
- msg_id
- msg_type
- from
- to
- topic
- qos
- timestamp
- status (delivered/failed/retried)
- retry_count
- delivered_at (如已投递)
- error (如失败)
```

日志位置: `logs/message-broker/audit-YYYYMMDD.log` (每日滚动, 不可删, Git 跟踪)。

## 8. 重试策略 (QoS 1+)

| 重试次数 | 退避 | 失败处理 |
|---|---|---|
| 1 | 100ms | 重试 |
| 2 | 500ms | 重试 |
| 3 | 2s | 重试 |
| 4 | 10s | 重试 |
| 5+ | - | 标记 failed, 进 Sponsor 通知 |

## 9. 业务项目接入 (5 步流程第 4 步)

业务项目接入 PMO 实例时:

1. 业务项目**注册**时声明订阅/发布主题 (`messaging.yaml`)
2. 业务项目通过 PMO 实例 API 发送/接收消息
3. 业务项目消息必须符合 PMO 协议 (本规则)
4. PMO 实例监管业务项目消息 (可监控 + 可审计)
5. 项目间消息进 PMO 指标看板 (Sponsor 可看)

## 10. 业务项目隔离 (m2.2 配合)

- 业务项目**只能订阅自己有权的主题** (PMO 校验)
- 业务项目**不能直接发消息给其他业务项目**, 全部经 broker
- 业务项目**不能修改消息元数据** (msg_id/timestamp/qos 由 broker 加)

## 11. 不参与业务

PMO Message-Broker-Agent **不参与业务内容**, 只管消息路由/可靠性/可监控/可审计。

- 业务负载 (`content` 字段) PMO 不解析
- 业务 schema 业务项目自定
- 业务流编排业务项目内约定 (m2.4)

## 12. 与 Sponsor 看板对接

m1.6 完成后, 项目间消息统计进 Sponsor 指标看板:

- 总消息数 / 成功率 / 延迟 (实时)
- 失败消息 (告警)
- 重试消息 (监控)
- 主题热度 (Top 10)

## 13. 与 M1.5 自检接入

m1.5 自检 16 项中, 加 1 项 m1.6 专门自检:

- **D17 消息流通自检**: 业务项目 A→broker→B 端到端可演示, QoS 校验, 重试可演示, 审计日志可查, 监控指标可看

## 关键决策

- **DEC-2026-0004**: 项目间消息流通机制 (2026-06-19, v0.11.0)
- 4 块关键设计: 协议 + 主题 + QoS + 监控审计
- 业务项目不直接通信, 全部经 PMO broker
- 消息可监控/可审计, 决策日志规范
- 影响: m2.5 跨边界契约加项目间消息契约
