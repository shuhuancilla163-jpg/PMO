# 0.0.16 业务元数据 3 项 (E1/E2/E3, DEC-2026-0005)

**元规则: 业务项目接入 PMO 必填的 3 项元数据** (m2.1, DEC-2026-0005)

## 1. 核心

业务项目接入 PMO 实例时, 必须提交 3 项元数据, 全部由 PMO 存, 业务项目可查:

- **E1 业务项目元数据** (register.yaml): 业务项目身份信息 (id/name/type/version/sponsor/状态/配额/隔离/合规)
- **E2 业务数据 schema** (data-schema.yaml): 业务项目数据模型 (业务项目定义, PMO 存, 业务流消息路由用)
- **E3 业务术语表** (glossary.yaml): 业务项目专业术语 (业务 agent 定义, PMO 验证一致性)

## 2. E1 业务项目元数据 (强制字段)

**路径**: `biz-projects/<biz_project_id>/register.yaml`
**存**: PMO 实例加载到 `config/biz-meta/<biz_project_id>.json`

```yaml
biz_project:
  id: "1.2-finance"           # 必填, 唯一
  name: "量化金融业务项目"     # 必填
  type: "finance-quant"        # 必填, 业务项目自定类型
  version: "0.1.0"             # 必填, semver
  sponsor: "Sponsor"           # 必填
  registered_at: "2026-06-19"  # 必填, ISO 8601

pmo_supervision:              # 必填
  enabled: true
  state: "active"              # registered/active/paused/blocked/completed/archived/canceled
  quota_4d:                    # 必填, 4 维
    token: "100K/day"
    time: "24/7"
    storage: "10GB"
    concurrency: "10"
  isolation_3d:                # 必填, 3 维
    data: "full"
    config: "independent"
    state: "independent"

agent_categories:             # 必填, 2 类 agent
  eng_roles:
    location: "eng-roles/"     # 5 阶段研发 agent (PMO 监管, 0.0.13)
    pmo_supervised: true
  biz_agents:
    location: "biz-agents/"    # 业务需求运营 agent (业务自管, 0.0.14)
    pmo_supervised: false

compliance_2_layers:          # 必填, 业务项目 2 层合规 (0.0.11)
  layer_1_overall: true       # 业务项目整体
  layer_2_eng_5_stages: true  # 业务项目内研发 5 阶段

m2_6_7_items:                 # 必填, m2.6 7 项
  registration: true
  state_machine: true
  quota_4d: true
  archive_4_levels: true
  isolation_3d: true
  alerting_3_levels: true
  checklist_6_plus_3: true
```

## 3. E2 业务数据 schema (业务项目定义, PMO 存)

**路径**: `biz-projects/<biz_project_id>/data-schema.yaml`
**存**: PMO 实例加载到 `config/biz-meta/<biz_project_id>-schema.json`

业务项目**自定**业务数据 schema, PMO 存 + 验证, 业务流消息路由时使用。

```yaml
biz_data_schema:
  version: "1.0"               # 必填, schema 版本
  defined_by: "biz_project"     # 必填, schema 由业务项目定义
  stored_by: "pmo_instance"    # 必填, PMO 实例存
  
  # 业务数据实体 (业务项目自定, 例如下面是金融业务项目示例)
  entities:
    - name: "order"            # 实体名
      fields:                  # 字段列表
        - name: "order_id"
          type: "string"
          required: true
        - name: "symbol"
          type: "string"
          required: true
        - name: "quantity"
          type: "number"
          required: true
        - name: "price"
          type: "number"
          required: true
        - name: "side"
          type: "enum"
          values: ["buy", "sell"]
          required: true
        - name: "timestamp"
          type: "datetime"
          required: true
      primary_key: ["order_id"]
    
    - name: "trade"
      fields:
        - name: "trade_id"
          type: "string"
          required: true
        - name: "order_id"
          type: "string"
          required: true
          foreign_key: "order.order_id"
        - name: "executed_price"
          type: "number"
          required: true
        - name: "executed_quantity"
          type: "number"
          required: true
        - name: "executed_at"
          type: "datetime"
          required: true
      primary_key: ["trade_id"]
  
  # 业务事件 (业务流路由用)
  events:
    - name: "order_placed"
      entity: "order"
      trigger: "new order"
    - name: "order_executed"
      entity: "trade"
      trigger: "order matched"
```

**PMO 验证规则**:
- 实体名唯一
- primary_key 字段在 fields 内
- foreign_key 引用必须存在
- type 必填, enum.values 必填
- 业务项目可任意扩展, PMO 不限制业务实体类型

## 4. E3 业务术语表 (业务 agent 定义, PMO 验证)

**路径**: `biz-projects/<biz_project_id>/glossary.yaml`
**存**: PMO 实例加载到 `config/biz-meta/<biz_project_id>-glossary.json`

业务项目**自定**业务术语, 业务 agent 用术语, PMO 验证术语一致性。

```yaml
biz_glossary:
  version: "1.0"
  defined_by: "biz_project"     # 必填
  validated_by: "pmo_instance"  # 必填
  
  # 业务术语 (业务项目自定)
  terms:
    - term: "order"
      definition: "交易订单, 业务核心实体"
      category: "entity"
      synonyms: ["委托", "单子"]
    - term: "trade"
      definition: "成交记录, 订单匹配后产生"
      category: "entity"
      synonyms: ["成交", "执行"]
    - term: "position"
      definition: "持仓, 业务核心状态"
      category: "state"
      synonyms: ["仓位", "敞口"]
    - term: "P&L"
      definition: "盈亏, 业务核心指标"
      category: "metric"
      synonyms: ["pnl", "盈亏"]
    - term: "drawdown"
      definition: "回撤, 风险指标"
      category: "metric"
      synonyms: ["回撤", "DD"]
  
  # 业务角色 (业务 agent 身份, DEC-2026-0003 业务自定)
  roles:
    - role: "Data-Engineer"
      identity: "金融数据工程师"
      scope: "数据采集/清洗/存储"
      uses_terms: ["order", "trade", "position"]
    - role: "Quant-Analyst"
      identity: "量化分析师"
      scope: "策略研发/信号生成"
      uses_terms: ["order", "P&L", "drawdown"]
    - role: "Risk-Manager"
      identity: "风险经理"
      scope: "风险监控/限额管理"
      uses_terms: ["position", "P&L", "drawdown"]
    - role: "Portfolio-Manager"
      identity: "组合经理"
      scope: "组合配置/调仓"
      uses_terms: ["position", "P&L"]
    - role: "Compliance-Officer"
      identity: "合规官"
      scope: "合规检查/审计"
      uses_terms: ["order", "trade"]
    - role: "Reporting-Analyst"
      identity: "报告分析师"
      scope: "业务报告/指标看板"
      uses_terms: ["P&L", "drawdown"]
```

**PMO 验证规则**:
- 业务项目业务 agent 必须用 E3 术语表中的术语
- 业务 agent role 名不能与 5 阶段研发 agent role 名重复 (避免混淆)
- 业务术语 category 必填 (entity/state/metric/action/role)
- 业务项目可任意扩展术语, PMO 不限制

## 5. 3 项元数据的关系

```
E1 业务项目元数据 (register.yaml)
   ↓ 提供
业务项目身份 (id/name/type/version/sponsor)
   ↓ 关联
E2 业务数据 schema (data-schema.yaml) — 业务项目用哪些数据实体
   ↓ 关联
E3 业务术语表 (glossary.yaml) — 业务项目用哪些术语, 业务 agent 身份
```

**业务项目接入 PMO 时**:
1. 业务项目填 E1 → PMO 加载
2. 业务项目填 E2 → PMO 存 + 验证
3. 业务项目填 E3 → PMO 存 + 验证
4. PMO 验证 E1.id 唯一, E2 字段引用 E3 术语, E3 role 不与 5 阶段研发 role 重复

## 7. 与 PMO 监管对接

| PMO 维度 | 消费元数据 |
|---|---|
| **维度 1: 业务项目整体** (L1 PMO-Main) | E1 (id/state/quota/isolation) |
| **维度 2: 业务项目内研发 5 阶段** (L2 Engineer-Agent) | E1.eng_roles + E3 (术语) |
| **维度 3: 业务项目内业务** (L2 Monitor-Agent) | E2 (data schema) + E3 (业务指标) |
| **项目间消息** (L2 Message-Broker) | E1.messaging + E2.events (业务事件路由) |

## 8. 业务项目接入顺序

业务项目接入 PMO 5 步流程的**前 3 步全部基于 m2.1**:

1. 业务项目注册 (E1) → PMO 加载元数据
2. 业务项目内研发 5 阶段声明 (E3.roles + eng_roles)
3. 业务项目内业务配置 (E2.data-schema + E3.terms + 业务自管)
4. 业务项目接入 PMO 消息机制 (E1.messaging + E2.events)
5. 业务项目接入 PMO 监管 (E1.pmo_supervision + E2.metrics)

## 9. 业务项目升级

业务项目 E1/E2/E3 变更:
- 业务项目 E1.version 升 → 业务项目整体升级
- 业务项目 E2.version 升 → 业务数据 schema 升级
- 业务项目 E3.version 升 → 业务术语表升级
- 业务项目每次升级触发 PMO 实例重新加载 (m2.2 业务不可变文档 + Git semver)

## 10. 不参与业务

PMO 存 E1/E2/E3 **元数据**, 不存业务项目**业务数据**:
- 业务数据 (订单/交易/持仓) 业务项目自存
- 业务流编排 (业务 agent 协作) 业务项目自管
- 业务异常定义 业务项目自定 (m2.4)
- 业务指标 5 项基础 + 业务项目自定义
