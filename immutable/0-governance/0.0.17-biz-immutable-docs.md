# 0.0.17 业务不可变文档 2 项 (F1/F5, DEC-2026-0006)

**元规则: 业务项目不可变文档规范 (m2.2, DEC-2026-0006)**

## 1. 核心

业务项目不可变文档 = 业务项目的**业务产出文档**, 与 PMO 治理不可变文档 (m0.1) 是两个独立层级:

- **PMO 不可变文档** (m0.1): PMO 治理层规范文档 (`immutable/0-governance/`)
- **业务不可变文档** (m2.2): 业务项目的业务产出文档 (`biz-projects/<id>/biz-docs/`)

## 2. F1 业务不可变文档骨架

**路径**: `biz-projects/<biz_project_id>/biz-docs/`
**性质**: 业务产出文档, 业务项目自维护, PMO 只验证 Git 归档完整性

### 4 类文档 (业务项目必须产出的核心文档)

```

│   └── nfr.md               # 非功能性需求 (性能/安全/可用性)
├── 02-design/               # 设计文档 (development 阶段产出)
│   ├── README.md
│   ├── biz-design.md        # 业务设计 (基于需求文档)
│   └── api-contract.md      # API 契约 (m2.5 跨边界契约前身)
├── 03-implementation/        # 实施文档 (development 阶段产出)
```

### 文档命名规范

| 文档类型 | 文件名格式 | 版本 | 不可变 |
|---|---|---|---|
| 需求文档 | `biz-requirements.md` | v1.0.0 | ✅ 发布后不可变 |
| 设计文档 | `biz-design.md` | v1.0.0 | ✅ 发布后不可变 |
| 业务流文档 | `biz-flow.md` | v1.0.0 | ✅ 发布后不可变 |
| API 契约 | `api-contract.md` | v1.0.0 | ✅ 发布后不可变 |
| 发布说明 | `release-notes.md` | v1.0.0 | ✅ 始终不可变 |

**关键**: 业务不可变文档以 Git commit 为时间戳, 以 semver 版本号标识, 不可覆盖只新增 (append-only 行为通过 Git 历史保证)。

## 3. F5 业务版本管理 (Git tag/release + semver)

**性质**: 业务项目的版本化管理, 与 PMO Git 独立

### 业务项目 Git 仓库

每个业务项目有一个独立 Git 仓库 (或在 PMO Git 内用目录隔离):
```
PMO Git/
└── biz-projects/
    └── <biz_project_id>/
        ├── .git/                    # 业务项目独立 Git 仓库
        ├── biz-docs/                # 业务不可变文档
        ├── biz-agents/             # 业务 agent (0.0.14)
        └── eng-roles/             # 5 阶段研发 agent (0.0.13)
```

**当前状态** (无独立 Git 仓库): 业务项目用 PMO Git, 通过 Git 历史保证不可变。

### semver 版本规范

```
v<MAJOR>.<MINOR>.<PATCH>

- MAJOR: 业务重大变更 (E1.id 不变, E2/E3 破坏性变更)
- MINOR: 新增功能 (E2 新增 entity/event, E3 新增 term/role)
- PATCH: 修复 / 文档更新 / 小调整
```

### Git tag 命名

```
biz.<biz_project_id>.v<MAJOR>.<MINOR>.<PATCH>

例如:
biz.1.2-finance.v1.0.0    # 初始发布
biz.1.2-finance.v1.1.0    # 新增 entity
biz.1.2-finance.v2.0.0    # 破坏性变更
```

### 版本发布流程

1. 业务项目完成一个研发周期
2. 业务 agent 编写 `biz-docs/` 文档
3. Git commit: `git add biz-docs/ && git commit -m "docs: <变更描述>"`
4. Git tag: `git tag biz.<id>.v<MAJOR>.<MINOR>.<PATCH>`
5. PMO 记录版本到 `config/biz-meta/<id>-versions.json`

### 版本历史查询

PMO 实例可查询业务项目版本历史:
```bash
git -C biz-projects/<id>/ log --oneline
git -C biz-projects/<id>/ tag -l "biz.<id>.*"
```

## 4. 业务不可变文档与 5 阶段的关系

| 阶段 | 产出文档 | Git commit | Git tag |
|---|---|---|---|
| requirement | `01-requirements/` | ✅ | - |
| development | `02-design/` + `03-implementation/` | ✅ | - |
| test | - (测试报告在 reports/) | ✅ | - |
| operations | `04-release/` | ✅ | ✅ (semver) |
| evaluation | 版本评审 | ✅ | ✅ (semver) |

## 5. 业务不可变文档与 m2.1 的关系

- **E2 data-schema** → 业务设计文档的基础 (`02-design/biz-design.md` 基于 E2 entities)
- **E3 glossary** → 需求文档的基础 (`01-requirements/biz-requirements.md` 基于 E3 terms)
- **m2.1 接入后** → 业务项目已有 E1/E2/E3, 可开始写 `biz-docs/`

## 6. 业务不可变文档与 PMO 监管对接

| PMO 维度 | 消费业务不可变文档 |
|---|---|
| **维度 2: 研发 5 阶段** (L2 Engineer-Agent) | 采集 biz-docs 文档完整性 (每阶段至少 1 篇) |

| **业务归档** (m2.6) | biz-docs 进入归档 (`archive/biz-projects/<id>/`) |
