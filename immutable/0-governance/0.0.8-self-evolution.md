# 0.0.8 自进化原则 (PMO 规范自进化 ≠ 模型自进化)

**Sponsor 平等讨论后修订, 不可变 (0.0.8)**

## 重要澄清

| 类型 | 是什么 | 谁做 | PMO 做不做 |
|---|---|---|---|
| **模型自进化** | 模型参数/训练数据/架构改 | 训练团队 (人+算力+合规) | **不做** |
| **治理自进化** | 规则/流程/工作流/状态机演进 | PMO 自身 (规则引擎) | **做** |
| **工程自改进** | 工具/方法/技术替换 | 人 + 工具市场 | **做** |
| **prompt 自改进** | 提示词改 | agent + 人 | **做** |

## PMO 自进化只做

- 治理自进化 (规则/流程/工作流演进, 触发: 复盘)
- 工程自改进 (工具/方法/技术替换, 触发: 新工具出现)

## PMO 不做

- 模型自进化 (模型参数/训练/架构改, 是模型团队的事)
- 训练数据自生成 (需要人+算力+合规审核)

## 适用范围 (2 层)

| 层级 | 自进化内容 | 触发 |
|---|---|---|
| **PMO 层** | 元规则演进, 流程演进, 阶段门控细节演进 | PMO 复盘 + 新工程工具 |
| **单项目层** | 项目规则演进, 项目反思, 项目最佳实践沉淀 | 单项目复盘 + 新业务工具 |

## 自进化触发 (双驱动)

| 触发源 | 说明 | 例子 |
|---|---|---|
| **复盘** | 什么有效, 什么无效, 什么卡, 哪里慢 | M6.1 自测复盘 → PMO 流程优化 |
| **新工具** | 新工程实现出现, 替换旧工程 | 新 agent 框架/Hermes 等出现 → PMO 工程实现层替换 |

## 自进化约束 (3 条, 不可变)

1. **不改价值层**: 5 价值 (高效/规范/可控/验收/优化) 永远不变
2. **不改治理层核心**: 阶段门控/业务状态/异常类型/反思维度等核心规则不变
3. **可改治理层细节 + 工程层**: 具体规则细节可改, 工程实现可换

## 模型选型策略 (不锁模型)

| 层 | 模型选型 |
|---|---|
| **价值层** | 不提具体模型 (价值不变) |
| **治理层** | 抽象为 "使用 LLM" (不锁定具体模型) |
| **工程实现层** | 具体模型 (GPT-4 / Claude / Hermes / 开源) 由工程实现层选 |

**原则**: 治理不绑定具体模型, 模型换了治理不变。

## Hermes 评估 (Sponsor 提到, 我的真实观点)

- **Hermes 是什么** (我需要查, 不能凭空说) — 可能是 NousResearch Hermes / Hermes Agent 框架 等
- **Hermes 是不是真有"自进化"** — 多数 agent 框架的"自进化"是 prompt/workflow 自改进, 不是模型自进化
- **Hermes 能不能用** — 取决于: 项目目标/合规/算力/部署/成本/能力
- **PMO 要不要锁 Hermes** — **不要** — Hermes 是工程实现层候选, 不是治理层强制
- **PMO 选不选 Hermes** — 由 **治理规则** 决定 (用 LLM), **工程层选具体哪个**
- **时机** — 工程实现层选型时评估, 不在治理层评估

**关键**: **Hermes 是工具, 不是治理** — 工程实现层考虑, 治理层不锁定。

## 不可变性

- 3 条约束**不可变** (改价值/核心治理规则)
- 具体规则细节可演进 (触发自进化)
- 自进化机制本身**不可变**
