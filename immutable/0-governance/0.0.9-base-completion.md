# 0.0.9 第一阶段基座完善原则 (Sponsor 提醒后补)

**不可变 (0.0.9)**

## 核心

第一阶段**无业务项目**, 目标: **完善 PMO 基座**, 避免后期接入多项目时改很多。

## 为什么第一阶段就要完善

| 原因 | 说明 |
|---|---|
| 后期改很多 = 风险 | 接入多项目时改基座, 影响所有项目 |
| 现在改 0 个项目 = 低风险 | 现在改基座, 没业务影响, 改错了重做 |
| 后期加项目 = 改基座 = 项目级联失败 | 多项目都依赖 PMO 基座, 基座动了项目都动 |
| 基座完善 = 后期接入快 | 业务项目按 PMO 接口接入, 不用改 PMO |

## 第一阶段目标 (基座完善清单)

| 维度 | 目标 | 任务 |
|---|---|---|
| **多租户** | PMO 支持 1+ 业务项目 (业务项目可同时跑) | m2.6 业务项目管理 (多租户) |
| **业务接入** | 业务注册, 业务状态, 业务配额, 业务归档 | m2.6 |
| **业务骨架** | 业务 agent 骨架, 业务 SOP 骨架, 业务文档骨架 | m2.3 + m2.4 + m2.1/m2.2 |
| **业务接入路径** | PMO↔业务接口契约 | m2.5 |
| **业务 checklist** | 业务启动 checklist + 业务监控 | m2.6 |
| **业务隔离** | 业务数据隔离, 业务配置隔离, 业务状态隔离 | m2.6 |

## 第一阶段不做 (等业务需求来)

| 不做 | 原因 |
|---|---|
| 业务 agent 具体实现 | 等 Sponsor 给业务需求, BA-Agent 接手 |
| 业务架构设计 | 等业务需求明确后做 |
| 业务流端到端 | 等业务 agent 实现后做 |
| 业务 MVP 验证 | 等业务流跑通后做 |

## 关键

第一阶段是 **"PMO 基座 + 业务骨架 + 业务接入路径"**, 不是 "PMO + 业务实施"。业务实施等下一阶段 Sponsor 介入后启动。

## 不可变性

- 基座完善原则**不可变**
- 第一阶段基座清单**不可变** (M0/M1/M2/M6/M7)
- 任何基座修改需 P9 决策 + Sponsor 拍板
